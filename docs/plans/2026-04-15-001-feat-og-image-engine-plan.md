# OG Image Engine + Hybrid Content Platform

## Overview

Build a typographic Open Graph image generation engine for cc4.marketing, replacing the Python/Gemini cover-generation scripts (`generate_cover*.py`) with a TypeScript-native solution that runs on Cloudflare Workers via `workers-og` (Satori + resvg-wasm). Images are brand-compliant: Righteous display, Outfit body, cream/rust/plum/mustard palette, hard shadows, thick borders — matching the site's retro-futuristic aesthetic.

**Delivery model is hybrid:**
- **Build-time pre-generation** for known, stable routes (17 module lessons, homepage, blog index, changelog, download, blog/authors) → PNGs written to `public/og/`
- **Runtime SSR endpoint** (`/og/blog/[slug]-[hash].png`) for Emdash-sourced blog posts → cached in R2 (`MEDIA` bucket) + Cloudflare edge cache

The content hash is embedded in the URL (cache-busting by URL change), so cache invalidation is automatic when post content changes — no webhooks or manual purges needed.

**Scope:** Phase 1 only (engine + dynamic endpoint). Phase 2 (Content Collections migration, Zod SEO enforcement) is explicitly out of scope (see brainstorm: `docs/brainstorms/2026-04-15-og-image-engine-brainstorm.md`).

---

## Problem Statement

### Current state

1. Blog covers are generated by two one-off Python scripts (`generate_cover.py`, `generate_cover_anthropic_growth.py`) that call Gemini 3 Pro via Vertex AI. Each new post requires editing the script, running it locally, and committing the PNG. **Vertex AI billing is currently disabled** (flagged in `.claude/skills/publish-post/SKILL.md:85` as of 2026-04-09), leaving a working fallback of hand-authored SVG + cairosvg.
2. All 17 module lessons and most static pages (homepage, blog index, changelog, download, authors) fall back to the generic `/og-image.png` or `/og-blog.png` in `BaseLayout.astro:44`. This is poor for social sharing — every page looks identical in link previews.
3. The `BaseLayout` OG contract is opaque: the `image` prop hardcodes `1200×630` dimensions (`BaseLayout.astro:139-140`) and absolutizes path against hardcoded `siteUrl`. There's no dynamic-OG URL pattern.
4. Emdash blog posts live in D1 (not in-repo), so build-time-only generation doesn't work — new posts published via `/publish-post` need covers but the build has already shipped.

### Desired state

- Every published page (static route + Emdash blog post) gets a distinctive, brand-compliant OG image automatically, without manual cover creation or Python script invocation.
- Manual override remains easy: set `post.featured_image.src` in Emdash and the engine yields.
- The `/publish-post` skill no longer needs to call Gemini/cairosvg for covers — the engine generates them.
- Developers can preview templates via a `/_dev/og-preview` route with live parameter tweaking and mock Facebook/Twitter/LinkedIn cards.
- Bundle size stays well under Cloudflare Workers' 3 MB free-tier compressed limit (paid plan has 10 MB; we target <2.5 MB total).

## Proposed Solution

**Library:** `workers-og` (kvnang) — a Cloudflare Workers-optimized wrapper around Satori + resvg-wasm that handles WASM initialization, font-cache boilerplate, and `ImageResponse`-style ergonomics. Alternatives considered below.

**Rendering strategy per content type:**

| Content type | Strategy | Delivery | OG URL pattern |
|---|---|---|---|
| Homepage, blog index, changelog, authors, download, brand-guide | Build-time → `public/og/` | Static asset | `/og/pages/home.png`, `/og/pages/changelog.png`, etc. |
| 17 module lessons | Build-time → `public/og/modules/` | Static asset | `/og/modules/{M}-{slug}.png` |
| Emdash blog posts | Runtime → `/og/blog/[slugHash].png` endpoint | R2 + edge cache | `/og/blog/{slug}-{hash8}.png` |
| Manual override | BaseLayout short-circuits | As-is | `post.featured_image.src` (existing behavior) |

**Caching:** Two layers.
- **R2 (`MEDIA` bucket, prefix `og/`):** durable source of truth. Keyed `og/{slug}-{hash}.png`.
- **`caches.default`:** edge cache, warmed via `ctx.waitUntil()` after successful R2 write.

**Cache invalidation:** No webhook needed. The content hash is embedded in the URL generated by `BaseLayout`. When post content changes, `BaseLayout` computes a new hash → new URL → crawler fetches fresh image. Stale R2 objects accumulate but are cheap; a monthly cron cleans them.

**Kill-switch:** `OG_ENGINE_ENABLED` env var (default `true`). When `false`, `BaseLayout` falls back to the existing `/og-blog.png` / `/og-image.png`. This is our deploy-free rollback if the engine misbehaves.

### Runtime endpoint

**`src/pages/og/blog/[slugHash].png.ts`:**

```ts
import type { APIRoute } from 'astro';
import { ImageResponse } from 'workers-og';
import { getEmDashEntry } from 'emdash';
import { env } from 'cloudflare:workers';
import { computeOgHash } from '../../../lib/og/hash';
import { renderBlogTemplate } from '../../../lib/og/templates/blog';
import { getFonts } from '../../../lib/og/fonts';
import { OG_DIMENSIONS } from '../../../lib/og/config';
import { MAGIC_PNG_BYTES } from '../../../lib/og/smoke';

export const prerender = false;

export const GET: APIRoute = async ({ params, request, url }) => {
  const match = (params.slugHash ?? '').match(/^(.+)-([a-f0-9]{8})$/);
  if (!match) return fallback404();

  const [, slug, urlHash] = match;

  // Layer 1: edge cache
  const cache = caches.default;
  const cacheReq = new Request(url.toString(), { method: 'GET' });
  const cached = await cache.match(cacheReq);
  if (cached) return cached;

  // Layer 2: R2
  const r2Key = `og/${slug}-${urlHash}.png`;
  const r2 = (env as any).MEDIA as R2Bucket;
  const r2Obj = await r2.get(r2Key);
  if (r2Obj) {
    const resp = pngResponse(await r2Obj.arrayBuffer());
    // Fire-and-forget edge cache warm. (Astro exposes ctx on locals.runtime.)
    // See "Caching" section below for the waitUntil pattern.
    return resp;
  }

  // Generate
  const { entry: post } = await getEmDashEntry('posts', slug);
  if (!post) return fallback404();

  // Validate URL hash matches current content hash (prevent serving stale content)
  const currentHash = await computeOgHash(post);
  if (urlHash !== currentHash) {
    // URL is outdated — redirect to current URL (edge-cached for a short time)
    return new Response(null, {
      status: 302,
      headers: {
        location: `/og/blog/${slug}-${currentHash}.png`,
        'cache-control': 'public, max-age=60',
      },
    });
  }

  const html = renderBlogTemplate({
    title: post.title,
    excerpt: post.excerpt,
    bylines: post.bylines ?? [],
  });

  const fonts = await getFonts();
  const imgResp = new ImageResponse(html, {
    width: OG_DIMENSIONS.width,
    height: OG_DIMENSIONS.height,
    fonts,
    format: 'png',
  });

  const png = new Uint8Array(await imgResp.arrayBuffer());
  if (!isValidPng(png)) return fallback200Generic();

  const resp = pngResponse(png);
  // waitUntil both R2 and edge cache; non-fatal on failure
  // (ctx.waitUntil pattern wired via Astro locals.runtime.ctx in actual impl)
  return resp;
};

function pngResponse(bytes: ArrayBuffer | Uint8Array): Response {
  return new Response(bytes, {
    headers: {
      'content-type': 'image/png',
      'cache-control': 'public, max-age=31536000, immutable',
      'cdn-cache-control': 'public, max-age=31536000',
    },
  });
}
```

**Key behaviors:**
- **404-safe:** missing post → 404 with short cache. Corrupt render → 200 with generic fallback PNG (so crawlers don't cache a 404).
- **Hash mismatch:** returns 302 to the current canonical URL with short TTL. Prevents serving stale content if a crawler hits an old URL.
- **Concurrency:** two simultaneous cold requests both render; last R2 write wins. Acceptable (Satori render ~80-200 ms is cheap).
- **R2 write failure:** non-fatal. Response still returns 200 with the rendered bytes; short `cache-control` so retry happens.

### Build-time pre-generation

**`src/lib/og/build.ts`:**

Invoked as a `prebuild` npm script:
```json
"scripts": {
  "prebuild": "tsx src/lib/og/build.ts",
  "build": "astro build"
}
```

Flow:
1. Iterate the 17 module lesson MDX files (read frontmatter: `title`, `module`, `lesson`, `description`, `duration`).
2. Iterate static pages: home, blog, changelog, authors, download, brand-guide. For each, emit a hardcoded config object with title/subtitle.
3. For each: render via the shared renderer; write to `public/og/{bucket}/{name}.png`.
4. Log bundle size stats.

Runs in ~20-30 seconds for 23 images. Because it runs *before* `astro build`, it can use plain Node.js (no Workers runtime constraints) — we use `@resvg/resvg-js` (Node-native, faster) rather than the WASM variant for this path.

**Why a separate Node path:** `@resvg/resvg-js` (Node) is 5-10× faster than the WASM variant and doesn't cost bundle size since it's only invoked at build time. `workers-og` is runtime-only.

### Astro config changes

```js
// astro.config.mjs
export default defineConfig({
  // ...existing config,
  vite: {
    plugins: [rawFonts(['.ttf'])],
    assetsInclude: ['**/*.ttf'],
  },
});

// Inline Vite plugin for loading .ttf as Uint8Array
function rawFonts(exts) {
  return {
    name: 'vite-plugin-raw-fonts',
    transform(_, id) {
      if (exts.some((e) => id.endsWith(e))) {
        const buffer = fs.readFileSync(id);
        return {
          code: `export default new Uint8Array([${buffer.join(',')}])`,
          map: null,
        };
      }
    },
  };
}
```

## System-Wide Impact

### Interaction Graph

When a crawler (e.g., Slack's link unfurler) fetches `https://cc4.marketing/blog/claude-code-for-marketing-guide-2026/`:

1. **Request hits Astro SSR handler** (`src/pages/blog/[slug].astro`, `prerender=false`).
2. Astro calls `getEmDashEntry('posts', slug)` → Emdash middleware reads D1 → returns Post record.
3. `<BaseLayout ogPost={post.data}>` renders.
4. BaseLayout calls `resolveOgImage(ctx, post)` → computes hash → returns `/og/blog/{slug}-{hash}.png`.
5. BaseLayout absolutizes → `<meta property="og:image" content="https://cc4.marketing/og/blog/...png">`.
6. Crawler parses HTML, sees og:image URL, fetches it.
7. **New request** hits `src/pages/og/blog/[slugHash].png.ts`.
8. Endpoint checks `caches.default.match(request)` — first hit miss.
9. Endpoint checks R2 `env.MEDIA.get('og/slug-hash.png')` — first hit miss.
10. Endpoint calls `getEmDashEntry('posts', slug)` again → re-reads D1 (cost: ~10 ms).
11. Endpoint validates `computeOgHash(post) === urlHash`. If mismatch: 302 to current URL.
12. Endpoint calls `renderBlogTemplate(post)` → HTML string.
13. Endpoint calls `new ImageResponse(html, { fonts, ... })` → `workers-og` invokes Satori (SVG) → resvg-wasm (PNG).
14. Endpoint builds `Response`; spawns two `waitUntil` tasks:
    - `env.MEDIA.put('og/slug-hash.png', png)` — R2 write.
    - `caches.default.put(request, response.clone())` — edge cache warm.
15. Endpoint returns the Response to the crawler.
16. Crawler caches the image per its own policy (Twitter: 7 days, FB/LinkedIn: ~30 days).

**Second request** (any subsequent fetch within edge cache TTL):
- Edge cache hits → Response returned in ~5-20 ms, no Worker CPU beyond the cache check.

**Content update flow:**
1. Editor updates post in Emdash → D1 `updatedAt` changes.
2. Next SSR request recomputes hash → new URL.
3. BaseLayout emits new `og:image` URL.
4. Crawler refetches (or was already serving from its own cache until TTL expiry).
5. Old R2 objects (`og/slug-<oldHash>.png`) become orphaned. Cheap to retain; cleaned by monthly cron (out of scope for v1).

### Error & Failure Propagation

| Failure point | Error | Handling | User-visible impact |
|---|---|---|---|
| Emdash D1 read fails | Thrown | Endpoint returns 500 (no fallback possible, no post data) | Crawler sees 500, retries later. Page HTML still serves (BaseLayout doesn't call engine synchronously) |
| `computeOgHash()` crypto fails | Unlikely (subtle crypto is available on Workers) | Throws → 500 | Same as above |
| `workers-og` render throws | From Satori internals (bad CSS, missing glyph) | Catch in endpoint → serve `/og-blog.png` fallback with 200 status (NOT 500, to avoid FB caching the error) | Crawler gets generic fallback, page still works |
| R2 `get()` fails | Network or quota | Catch → proceed to render path | Slightly slower cold request, no user impact |
| R2 `put()` fails inside waitUntil | Network or quota | Logged, no retry; response already sent | Next request re-renders (edge cache still warms) |
| `caches.default.put()` fails | Rare | Logged; response already sent | Slightly more R2 reads next time |
| Font TTF bytes corrupted | Boot-time failure | Worker fails to deploy (caught in Phase 0 spike) | Rollback to previous deploy |
| WASM compile fails (Astro #15684) | Cold-start failure | Dev-time catch during Phase 0 spike | Must keep `prerender=false` |

**Kill-switch exercise path:** Set `OG_ENGINE_ENABLED=false` in Cloudflare dashboard. Next request:
- BaseLayout sees `engineEnabled=false` → returns `/og-blog.png` (existing static asset).
- No endpoint invocation. No R2 read. No rendering.
- Rollback time: ~30 seconds from dashboard click to effect.

### State Lifecycle Risks

1. **Orphaned R2 objects.** When post `updatedAt` changes, the old hash URL is abandoned. R2 objects accumulate. At current cadence (~10 posts/year × ~5 edits/post × ~100 KB), ~5 MB/year — negligible. Monthly cron cleanup is *future work*, not blocking.
2. **Stale edge cache after kill-switch toggle.** If engine is disabled, existing `/og/blog/...` URLs may still be in crawler caches. Fine — worst case, crawler fetches from edge cache for remainder of TTL, then hits the endpoint which serves the fallback. No data loss.
3. **Partial R2 write.** Impossible — R2's `put()` is atomic. Either the object exists completely or not at all.
4. **Concurrent regeneration (two cold crawlers).** Both render; both write to R2; last-write-wins. Results are byte-identical (deterministic template + same inputs). No lifecycle risk.
5. **Template version bump (`OG_TEMPLATE_VERSION += 1`).** Invalidates all hashes globally. Crawlers with cached old URLs see 302→new URL → hit endpoint cold → regenerate all active posts on demand. Old R2 objects orphaned. Expected behavior.

### API Surface Parity

The engine is invoked from **three entry points** that must stay in sync:

| Entry | Uses | What computes the hash |
|---|---|---|
| `BaseLayout.astro` (SSR render time) | `resolveOgImage(ctx, post)` | `src/lib/og/hash.ts` — same impl |
| `src/pages/og/blog/[slugHash].png.ts` (endpoint) | Validates URL hash against current content hash | Same `src/lib/og/hash.ts` |
| `src/lib/og/build.ts` (build time) | Generates static PNGs | Doesn't hash — static content |

The shared module `src/lib/og/hash.ts` is the single source of truth. Any change to the hash algorithm (field set, encoding, canonicalization) must ripple through all three. Enforce via a shared unit test that pins the hash output for a known input.

**JSON-LD `ImageObject` (BaseLayout + [slug].astro):** Hardcoded `1200×630`. If dimensions change, update both places — or better, centralize in `src/lib/og/config.ts` and import. Phase 3 does the centralization.

### Integration Test Scenarios

1. **Cold boot, first request for a new post.** Publish a new post via `/publish-post`. Immediately share to Slack. Expect: within 500 ms, Slack fetches `/og/blog/new-post-{hash}.png`, engine renders, R2 persists, Slack preview shows correct title + byline.
2. **Content update invalidation.** Edit an existing post's title via Emdash. Visit the blog page. Confirm `<meta og:image>` now contains a new hash. Fetch the old URL — expect 302 to new URL. Fetch new URL — expect fresh PNG.
3. **Kill-switch activation.** Set `OG_ENGINE_ENABLED=false`. Visit any blog page. Confirm `<meta og:image>` is `https://cc4.marketing/og-blog.png`. Fetch an old dynamic URL directly — expect 302 to `/og-blog.png`.
4. **Crawler with restrictive UA.** Simulate Facebook's crawler (`curl -A "facebookexternalhit/1.1"`). Expect PNG return, not 403/blocked. Validate at https://developers.facebook.com/tools/debug/ for each live blog post.
5. **Missing post (404).** Hit `/og/blog/nonexistent-slug-12345678.png`. Expect 404 with `cache-control: public, max-age=300` (short cache to avoid FB pinning a 404 indefinitely). Alternative: return 200 with generic fallback — decided against because a 200 here is misleading to crawlers.

## Alternative Approaches Considered

### A. Raw Satori + `@resvg/resvg-wasm` (DIY)
- **Pros:** No wrapper dependency; full control; slightly smaller bundle.
- **Cons:** Must hand-roll WASM init, font fetch/cache helpers, emoji shims. That's ~150-300 LOC of boilerplate kvnang already wrote and maintains.
- **Rejected:** No meaningful win over `workers-og`. Revisit only if `workers-og` becomes unmaintained.

### B. Cloudflare Browser Rendering (headless Chromium)
- **Pros:** Full CSS support (no Satori flexbox-only limits). Supports any HTML page as a screenshot source. Official Cloudflare guide exists for Astro.
- **Cons:** Paid plan required. Orders of magnitude slower (~2-5 sec cold). Higher cost per render.
- **Rejected:** Overkill for typographic layouts. Keep in back pocket if template design ever needs features Satori can't render.

### C. Build-time only (no runtime endpoint)
- **Pros:** Zero Worker bundle impact. Maximum cache-ability.
- **Cons:** Breaks the Emdash publish flow — new posts have no cover until a full site redeploy. Would require a GitHub Action webhook wired to Emdash admin "publish" button.
- **Rejected at hybrid stage:** Hybrid model gives us the speed of build-time for stable routes plus the flexibility of runtime for Emdash. Pure build-time stays as the rollback strategy if WASM bundle exceeds limits.

### D. Keep Gemini illustrations as the blog cover style
- **Pros:** Editorial illustration feel preserved.
- **Cons:** Requires billing, not deterministic, no text on images means every cover looks similar in link previews, Python scripts are brittle.
- **Rejected explicitly by user** — Phase 0 brainstorm chose typographic templates (see brainstorm: `docs/brainstorms/2026-04-15-og-image-engine-brainstorm.md` — choice **x**).

### E. Content Collections migration for posts (full Phase 2 from BiPlus plan)
- **Pros:** Zod SEO enforcement, type-safe front matter, auto-wired OG.
- **Cons:** Migrates blog posts out of Emdash, which is the CMS the user chose.
- **Rejected explicitly by user** — scope was `a+b` (engine + dynamic endpoint), not `c` (platform migration). See brainstorm § Scope Cuts.
